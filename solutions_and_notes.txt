1. CHOPPY AUDIO — BLUETOOTH INTERFERENCE
================================================================================

PROBLEM:
  The Raspberry Pi 5 has a single shared antenna for both Wi-Fi and Bluetooth.
  When you have multiple BT peripherals connected (mouse, keyboard) AND are
  streaming BT audio to an output speaker, all those devices compete for the
  same Bluetooth radio bandwidth and timeslots. Bluetooth Classic audio (A2DP)
  is particularly bandwidth-hungry compared to BT HID devices (mice/keyboards),
  but having several devices all contending for the same 2.4 GHz radio causes
  packet retransmissions, scheduling delays, and ultimately choppy/stuttering
  audio.

  Additionally, Wi-Fi (especially 2.4 GHz) operates in the same frequency band
  as Bluetooth, adding further interference.

ROOT CAUSE:
  - The Pi 5's onboard BT controller is shared with Wi-Fi on one antenna.
  - BT Classic can only communicate with one device at a time using time-division
    multiplexing (piconet scheduling). More devices = more scheduling contention.
  - A2DP audio streaming needs consistent, timely delivery of audio packets.
    When a mouse or keyboard poll interrupts or delays an audio packet, you
    hear a skip or stutter.

SOLUTIONS (no code changes):

  a) Use a USB Bluetooth Dongle for Audio
     - Buy a USB BT 5.0+ dongle (e.g., TP-Link UB500, ASUS USB-BT500).
     - Plug it into the Pi 5. This gives you a SECOND Bluetooth controller.
     - Configure BlueZ to use the USB dongle for audio connections and the
       onboard adapter for peripherals (or vice versa).
     - This completely eliminates the contention because audio and HID traffic
       are on separate radios.
     - To list adapters:  bluetoothctl list
     - To select a specific adapter:  bluetoothctl select <MAC>
     - Then pair/connect audio devices on that adapter.

  b) Use Wired or USB RF Dongle Peripherals
     - Switch your mouse and keyboard to USB (wired or USB RF dongle).
     - This frees up the onboard BT radio entirely for audio streaming.
     - Simplest and cheapest solution if you have USB peripherals available.

     NOTE ON USB RF DONGLES (e.g., Logitech Unifying, Razer USB receivers):
     - Peripherals that come with their own small USB RF dongle do NOT use
       Bluetooth. They use a proprietary 2.4 GHz RF protocol with their own
       independent radio chip on the dongle.
     - Because they bypass the Pi's onboard Bluetooth controller entirely,
       they do NOT compete for BT timeslots and will NOT cause the same
       contention/choppy audio problem.
     - They do still operate in the 2.4 GHz band, so there is a theoretical
       possibility of minor RF interference in very crowded environments, but
       this is negligible compared to BT piconet scheduling contention.
     - Bottom line: USB RF dongle peripherals are effectively the same as
       wired for this purpose — they free up the Pi's BT radio for audio.

  c) Switch Wi-Fi to 5 GHz Band
     - If you're using Wi-Fi, make sure your router is set to 5 GHz.
     - 2.4 GHz Wi-Fi directly interferes with Bluetooth (same frequency band).
     - On the Pi, you can force 5 GHz in /etc/wpa_supplicant or nmcli:
         nmcli connection modify <name> 802-11-wireless.band a

  d) Adjust the Bluetooth Audio Codec
     - SBC (the default A2DP codec) is more sensitive to packet loss.
     - If your BT speaker supports it, try AAC or aptX via PulseAudio config.
     - In /etc/pulse/default.pa or via pactl, you can load
       module-bluetooth-discover with specific codec arguments.
     - However, codec support depends on the remote device.

  e) Reduce Bluetooth Polling Rates
     - BT mice often poll at high rates (125 Hz+). Some let you reduce this
       in their companion software/firmware, which frees up more timeslots
       for audio.

  RECOMMENDED: Option (a) or (b). A USB BT dongle dedicated to audio is the
  most robust fix. Wired peripherals is the simplest.

2. AUDIO BUFFERING FOR SMOOTH INPUT
================================================================================

PROBLEM:
  Even after resolving BT interference, Bluetooth audio inherently has jitter —
  packets don't arrive at perfectly regular intervals. Without buffering, each
  tiny timing variation translates directly into an audible glitch. A buffer
  absorbs these variations by accumulating a small reservoir of audio data
  before playback begins (or before forwarding to the output sink).

HOW BUFFERING WORKS:
  Think of it like a water tank between a variable-flow pipe (BT input) and a
  faucet that must flow at a constant rate (audio playback). The tank absorbs
  surges and dips so the faucet stays steady.

  - Audio arrives from BT in chunks (packets). Sometimes a packet is 2 ms late,
    sometimes 5 ms early.
  - A buffer of, say, 50-100 ms holds enough data that even if a packet is
    delayed, there's still audio to play while waiting.
  - Trade-off: larger buffer = smoother audio but more latency (delay between
    what's happening on the phone and what you hear).

WHERE TO CONFIGURE BUFFERING:

  a) PulseAudio daemon.conf (system-wide buffer settings)
     - File: /etc/pulse/daemon.conf
     - Key settings:
         default-fragment-size-msec = 25     (size of each audio chunk)
         default-fragments = 4               (number of chunks in buffer)
     - Total buffer = fragment-size × fragments = 25 × 4 = 100 ms
     - For BT audio, try: fragment-size = 25, fragments = 4 to 8
     - Restart PulseAudio after changes: pulseaudio -k && pulseaudio --start

  b) PulseAudio tsched (Timer-Based Scheduling)
     - By default, PulseAudio uses timer-based scheduling which dynamically
       adjusts buffer sizes. For BT audio this can sometimes be too aggressive.
     - You can increase the tsched buffer in /etc/pulse/default.pa:
         load-module module-udev-detect tsched_buffer_size=65536
     - This gives PulseAudio a larger runway for scheduling.

  c) Environment Variable: PULSE_LATENCY_MSEC
     - Set before launching your application:
         export PULSE_LATENCY_MSEC=100
     - This requests PulseAudio to use at least 100 ms of buffering.
     - Quick way to test different buffer sizes without editing config files.

  d) Application-Level Buffer (in your code, when you're ready)
     - You could implement a ring buffer in AudioManager that collects audio
       samples from the BT source and drains them at a steady rate to the
       output sink.
     - PulseAudio's built-in module-loopback already does this:
         pactl load-module module-loopback source=<bt_source> sink=<usb_sink> \
           latency_msec=100 adjust_time=3
     - module-loopback creates a bridge between a source and sink with its
       own configurable internal buffer. The "latency_msec" parameter controls
       the buffer size directly.
     - This is likely the best approach for your use case — it gives you
       per-stream buffering without writing buffer code yourself.

  RECOMMENDED: Start with module-loopback (latency_msec=100) and adjust the
  value up/down. Also set default-fragment-size-msec and default-fragments in
  daemon.conf as a baseline.

3. GUI NOT SWITCHING AUDIO INPUT
================================================================================

PROBLEM:
  When you select a source in the GUI and click "Set Input", the status bar
  says it succeeded, but the audio doesn't actually switch. You still hear
  from the original source.

ROOT CAUSE:
  The code calls:
      pactl set-default-source <source_name>

  This command only changes what PulseAudio uses for NEW recording streams.
  It does NOT move EXISTING streams (called "source-outputs") that are already
  recording from another source. If you have an active loopback, recording
  app, or any stream already connected to the old source, it will stay on
  the old source.

  This is how PulseAudio works by design — set-default-source is a "going
  forward" setting, not a "move everything now" command.

  The same issue applies to set-default-sink for output — existing playback
  streams (sink-inputs) won't move either.

SOLUTION:
  After setting the default source, you also need to move all existing
  source-outputs to the new source. The commands are:

  For inputs (sources):
      pactl list short source-outputs
      — This lists active recording streams with their IDs.

      pactl move-source-output <stream_id> <new_source_name>
      — This moves a specific recording stream to the new source.

      To move ALL source-outputs:
        for each source-output ID from "pactl list short source-outputs":
            pactl move-source-output <id> <new_source_name>

  For outputs (sinks), the equivalent is:
      pactl list short sink-inputs
      pactl move-sink-input <stream_id> <new_sink_name>

  So the full "switch source" operation should be:
      1. pactl set-default-source <name>       (for future streams)
      2. pactl list short source-outputs        (get active stream IDs)
      3. pactl move-source-output <id> <name>   (for each existing stream)

  And for "switch sink":
      1. pactl set-default-sink <name>
      2. pactl list short sink-inputs
      3. pactl move-sink-input <id> <name>      (for each existing stream)

  In the code, set_default_source() in audio_manager.py would need to also
  enumerate and move source-outputs, and set_default_sink_by_name() would
  similarly need to move sink-inputs.

  Additionally, if you're using module-loopback to bridge BT input to USB
  output, switching the source means unloading the old loopback module and
  loading a new one pointed at the new source — OR using move-source-output
  on the loopback's recording stream.

4. NON-LOGICAL SOURCE NAMES
================================================================================

PROBLEM:
  PulseAudio assigns technical internal names to sources:
  - The first BT device connected gets a sink, and PulseAudio automatically
    creates a ".monitor" source for that sink. A monitor source captures
    whatever audio is playing on the sink (like a loopback). So you see
    something like:
        bluez_sink.XX_XX_XX_XX_XX_XX.a2dp_sink.monitor
  - The second BT device shows up with a proper source name like:
        bluez_source.YY_YY_YY_YY_YY.a2dp_source
  - Neither of these tells you the actual device name (e.g., "John's iPhone").

  The ".monitor" source is NOT the actual microphone/audio input from the BT
  device — it's PulseAudio mirroring the sink's output. This is an important
  distinction: if a phone is sending audio via A2DP, it creates a SINK (not a
  source). The .monitor on that sink lets you "record" what's playing, but
  it's not a true audio input.

WHY THIS HAPPENS:
  PulseAudio names are derived from the BlueZ D-Bus object paths and the
  Bluetooth MAC addresses. PulseAudio does store a "device.description"
  property that often contains the friendly name, but the "name" field
  (which is what your code displays in the GUI) is always the technical
  identifier.

SOLUTION:
  The friendly device name IS available — it's just in a different property.
  When you run:
      pactl list sources
  
  Each source has a "Properties" section that includes:
      device.description = "My iPhone"
      device.alias = "My iPhone"
      bluetooth.alias = "My iPhone"

  The "Description:" line at the top of each source entry usually contains
  the friendly name too.

  Your _parse_sources() and _parse_sinks() methods already extract the
  "Description:" field but get_sources() and get_sinks() don't include it.

  Currently, get_sources() returns:
      [{"id": s["index"], "name": s["name"]}]

  It should also return the "description" field and the GUI should display
  that instead of (or alongside) the raw name.

  So the fix is:
      1. In get_sources() and get_sinks(), include the "description" key
         that _parse_sources()/_parse_sinks() already extract.
      2. In the GUI, display the description in the listbox instead of the
         raw PulseAudio name.
      3. Continue using the raw "name" internally for pactl commands (since
         pactl requires the technical name, not the description).

  For even more control, you could parse the "Properties:" block in
  _parse_sources() to extract "bluetooth.alias" or "device.description"
  which sometimes give cleaner names than the top-level Description field.

  Alternatively, you can rename sources/sinks manually via:
      pactl set-source-name <source_index> <new_name>        (if supported)
  or update the description:
      pactl update-source-proplist <source_name> device.description="My Phone"

  But the simplest approach is to display the already-available Description
  field in the GUI while keeping the raw name for commands.
